#### 复现

```

```
TypeError: Cannot read properties of undefined (reading 'on')

### 1.2 问题二：respawn 血量恢复链路

#### 复现
```
npx playwright test tests/demo-playthrough-audit.spec.mjs --grep "Player death and respawn"
```
基线日志：
```
Starting HP: 3/3
After kill: HP=0
Respawn button active: true
After respawn: HP=0/3            ← 按 TouchRespawn 后 HP 仍为 0
[CRITICAL] Respawn did not restore player health
```

#### 最小根因
死亡链路与手动按钮链路对“恢复 HP”的责任没有统一：

- **自动死亡路径**：`HealthComponent.applyDamage` → `HEALTH_EVENT_DEPLETED` → `PlayerController.onHealthDepleted` → `GameManager.requestRespawn()`（仅 emit + `playerRoot.setWorldPosition`）→ 然后 `PlayerController.respawnAt(checkpoint)` 才真正 `health.resetFull()`；如果 `GameManager.checkpoint` 为空就直接 `return`，HP 不重置。
- **手动按钮路径**：`TouchCommandButton`（Respawn）和 `KeyboardInputDriver`（Key R）都只调用 `GameManager.requestRespawn()`，没有任何代码再调 `respawnAt()`，所以 HP 永远不会恢复。

也就是说，HP 重置依赖于 `onHealthDepleted` 内部的第二次手动调用 `respawnAt(checkpoint)`，但手动按钮链路完全没有走这条分支，自然不会重置。

#### 最小修改
1. [assets/scripts/player/PlayerController.ts](/E:/cv5/wisdom/assets/scripts/player/PlayerController.ts) — 让 PlayerController 监听 `GAME_EVENT_RESPAWN_REQUESTED`：
   - 新增 `onEnable` / `start` 中 `bindGameManager()`（参考 `BreakableTarget` 模式，应对 GameManager onLoad 比 PlayerController 慢的情况）。
   - 新增 `onRespawnRequested(checkpoint)`：根据 checkpoint（或当前位置作为兜底）调用 `respawnAt()`，统一重置位置 + HP + attackTimer + forcedMove。
   - 把 `onHealthDepleted` 简化为只调 `GameManager.requestRespawn()`，让事件统一驱动。
   - 在 `onDisable`/`onDestroy` 同步 `events.off`。

   修改后两条路径完全一致：自动死亡和手动 TouchRespawn 都走 `GameManager.requestRespawn()` → emit `GAME_EVENT_RESPAWN_REQUESTED` → `PlayerController.onRespawnRequested` → `respawnAt()`。位置由 `GameManager.playerRoot.setWorldPosition` 提供兜底（与原 setWorldPosition 路径并存，幂等）。

2. [tests/helpers/playwright-cocos-helpers.mjs](/E:/cv5/wisdom/tests/helpers/playwright-cocos-helpers.mjs) — `killPlayer` 增加 `{ disableAutoRespawn?: boolean }` 选项。当 true 时，先临时 `events.off` 解绑 PlayerController 的 health-depleted 与 respawn-requested 监听，跑完 kill 循环后再 `events.on` 复原。仅供需要观察“已死状态”的隔离测试使用，默认行为不变（仍会在循环里反复触发 respawn）。

3. [tests/checkpoint-respawn.spec.mjs](/E:/cv5/wisdom/tests/checkpoint-respawn.spec.mjs) — `player health at zero triggers depleted state` 改为 `disableAutoRespawn: true`（保持原意：测 depletion 机制本身）。新增一个用例 `manual respawn restores player health and clears combat state`，显式验证生产路径下 HP 由 `GameManager.requestRespawn()` 恢复。

4. [tests/demo-playthrough-audit.spec.mjs](/E:/cv5/wisdom/tests/demo-playthrough-audit.spec.mjs) — `Player death and respawn`：
   - `killPlayer` 改为 `disableAutoRespawn: true`，先观察“已死”
   - 把原本的 `console.log('[CRITICAL] Respawn did not restore player health')` 软失败改成 `expect(respawnWorked).toBe(true)` 的硬断言，确保未来这个 bug 回归会直接让套件红
   - `respawnWorked` 现在判断 HP 必须等于 max（不是仅仅 > 0）

5. [tests/error-recovery.spec.mjs](/E:/cv5/wisdom/tests/error-recovery.spec.mjs) — `BUG#2: player at 0 HP with no checkpoint does not crash` 同样改 `disableAutoRespawn: true`，保持“非崩溃”的保护意图，同时不依赖 bug 行为。

注意：本轮**不**修改测试是为了让结果绿，而是为了把测试与新的、正确的产品行为对齐：原测试隐含的“死亡停在 HP=0”假设本身就是这个 bug 的产物；修复 bug 后该假设不再成立，需要拆成两类用例来分别验证“depletion 机制”和“unified 回血”，反而提升了对 demo 目标的保护。

### 1.3 问题三：`FieldWest` / `FieldRuins` 控制台 critical

#### 复现
```
npx playwright test tests/demo-playthrough-audit.spec.mjs --grep "FieldWest -- full audit|FieldRuins -- full audit"
```
基线日志：
```
[CRITICAL] FieldWest: JS console error: Failed to allocate chunk in StaticVBAccessor, the requested buffer might be too large: 150336 bytes
[CRITICAL] FieldRuins: JS console error: Failed to allocate chunk in StaticVBAccessor, the requested buffer might be too large: 160704 bytes
```

#### 最小根因
两个场景的 backdrop 用 `SceneDressingSkin`（TILED 模式）把 32×32 的地表纹理铺到 `FieldBackdrop`（1840×560） 和 `RuinsBackdrop`（1960×560） 上，单次绘制需要：

- FieldWest：ceil(1840/32) × ceil(560/32) = 58 × 18 = 1,044 tiles → 4,176 vertices → 150,336 bytes（vfmtPosUvColor 36 bytes/vertex）
- FieldRuins：62 × 18 = 1,116 tiles → 4,464 vertices → 160,704 bytes

Cocos 的 2D batcher 默认 `cc.macro.BATCHER2D_MEM_INCREMENT = 144`（KB），即每个 `StaticVBAccessor` 单次最多分配 `floor(144*1024/36) = 4,096 vertices`。1,044 / 1,116 tile 都恰好越线，导致 `allocateChunk` 直接拒绝并打印 critical（同时该批 sprite 静默不绘制 — 看到的是“红错 + 该 backdrop 绘制失败”）。

`cc.macro.BATCHER2D_MEM_INCREMENT` 是引擎为这种"批处理预算"开放的官方参数（注释里明确允许调到最大 2303KB），把它调到 256KB 后单次可分配 `floor(256*1024/36)=7,281 vertices`，足够覆盖以上两个 backdrop 以及未来类似规模。

#### 关键陷阱
最初在 `PhysicsBootstrap.onLoad` 里设 macro 不生效：通过 `chromium.evaluate` 抓 `batcher2D._bufferAccessors` 才发现 `_vCount = 4096`（旧值）。Cocos 的 `Batcher2D.switchBufferAccessor` 在第一次拿同 stride 的 accessor 时 `new StaticVBAccessor(device, attributes)`，构造里就把 `_vCount = vCount || floor(macro * 1024 / vbStride)` **冻结**了，之后改 macro 不会重算。`PhysicsBootstrap.onLoad` 在场景生命周期中执行，但已经晚于第一次 sprite render（preview 加载阶段就已经画了若干 sprite），所以 macro 改值没有及时落地。

正确做法：把 macro 设置放在脚本**模块顶层**（不是 onLoad）。Cocos 加载所有 `@ccclass` 脚本作为类注册环节，比第一帧 render 更早；模块顶层的赋值就在那一刻生效。验证：再次 `evaluate(batcher2D._bufferAccessors)` 看到 `_vCount = 7281`。

#### 最小修改
[assets/scripts/core/PhysicsBootstrap.ts](/E:/cv5/wisdom/assets/scripts/core/PhysicsBootstrap.ts) — 在文件顶层（模块加载时执行）写入：

```ts
const RECOMMENDED_BATCHER2D_MEM_INCREMENT_KB = 256;
if ((macro.BATCHER2D_MEM_INCREMENT ?? 0) < RECOMMENDED_BATCHER2D_MEM_INCREMENT_KB) {
  macro.BATCHER2D_MEM_INCREMENT = RECOMMENDED_BATCHER2D_MEM_INCREMENT_KB;
}
```

注释里写清来龙去脉与"必须在模块顶层"的原因，避免后人误把它挪到 onLoad。`PhysicsBootstrap` 选作宿主的原因是 9 个主流程场景全都引用了它，模块一定会被 Cocos 在启动期加载。

不动任何场景文件、不动 `SceneDressingSkin`、不动 backdrop 尺寸——这是 Cocos 引擎自己开出来的旋钮，调整属于"使用引擎的官方接口"，不是绕过控制台错误。

### 1.4 问题四：主流程 `DebugLabel` 清理

#### 复现
```
rg -n "DebugLabel" assets/scenes assets/prefabs assets/scripts
npx playwright test tests/demo-playthrough-audit.spec.mjs --grep "full audit|No placeholder labels visible"
```
基线状态：
- 静态：`MechanicsLab.scene` 包含 `DebugLabel/DebugLabel-Visual/DebugLabel-Label`；`EchoBox.prefab`、`EchoSpringFlower.prefab`、`ArrowProjectile.prefab` 各自都有 `DebugLabel` 子节点
- 运行时：FieldWest（放下 EchoBox 后）/ DungeonRoomB（放下 SpringFlower）/ MechanicsLab 的 audit 都 `[COSMETIC] Debug elements visible: DebugLabel...`

#### 最小根因（双层）
1. **静态层**：上述 4 个资产里 DebugLabel 节点 `_active = true`。生成器源头（`generate-mechanics-lab.mjs`）在 `addPrefabLabel` 与 `addLabeledNode('DebugLabel', ...)` 处都用了默认 `active=true`。
2. **运行时层（更核心）**：`SpriteVisualSkin.setPlaceholderLabelVisible(node, visible)` 会**遍历所有带 Label 组件的子节点并强制 `child.active = visible`**。在 placeholder 模式（`hideLabelWhenSkinned=true` 但还没绑定 spriteFrame）下 `visible=true`，所以即使我把 prefab 里的 `DebugLabel._active` 改成 false，运行时一进 onLoad 又会被点亮。这是 prefab 里设了 `_active: false` 仍然在 runtime 上被打开的真正原因，靠静态修复无法根治。

#### 最小修改

| 文件 | 改动 |
|---|---|
| [assets/prefabs/EchoBox.prefab](/E:/cv5/wisdom/assets/prefabs/EchoBox.prefab) | DebugLabel 节点 `_active: true → false` |
| [assets/prefabs/EchoSpringFlower.prefab](/E:/cv5/wisdom/assets/prefabs/EchoSpringFlower.prefab) | 同上 |
| [assets/prefabs/ArrowProjectile.prefab](/E:/cv5/wisdom/assets/prefabs/ArrowProjectile.prefab) | 同上 |

| [tools/generate-mechanics-lab.mjs](/E:/cv5/wisdom/tools/generate-mechanics-lab.mjs) | `addPrefabLabel` 内增加 `startsInactive = name === 'DebugLabel'` 分支，把 DebugLabel 类型的节点默认设为 inactive；MechanicsLab 中 `addLabeledNode('DebugLabel', ...)` 那行把第 9 个参数 `active` 由 `true` 改成 `false` |
