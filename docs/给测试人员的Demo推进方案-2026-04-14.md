# 给 Claude 执行的 Demo 收口作战手册（2026-04-14）

## 1. 文档用途

这不是给人类讨论方向用的“概念方案”，而是一份给 Claude 直接执行的作战手册。

目标只有一个：

- 把当前项目收口成一个可以交给测试人员独立游玩的内部 demo

这里的“可以交给测试人员”指的是：

- 测试人员不需要开发者陪跑
- 测试人员能自己进入、推进并完成当前主流程
- 测试人员遇到的问题主要是“版本问题”，而不是“版本根本不可测”
- 测试人员提交的 bug 能够稳定复现到同一版 demo

这份文档默认 Claude 有足够 token，可以进行大规模阅读、重复验证、完整回归，不允许因为“节省 token”而跳过关键检查。

## 2. Claude 的唯一目标

Claude 在这个任务里只允许追求下面三件事：

1. 修掉当前影响试玩的关键问题
2. 把当前主流程收成“无陪跑可玩”
3. 把版本收口到可以稳定发给测试人员

Claude 不允许在这个阶段追求：

- 新功能
- 新系统
- 更大内容量
- 额外扩图
- 体系化重构
- 纯审美型 polish

如果某个改动不能直接帮助 demo 变得“更稳定、更易懂、更可分发”，默认不做。

## 3. Demo 成功定义

当前固定 demo 范围为：

- `StartCamp`
- `FieldWest`
- `FieldRuins`
- `DungeonHub`
- `DungeonRoomA`
- `DungeonRoomB`
- `DungeonRoomC`
- `BossArena`

成功定义如下：

- 主流程从 `StartCamp` 到 `BossArena` 可以跑通
- 死亡、重生、存档、读档不会破坏主线
- 触屏主要操作可用
- 不出现明显红错、失效按钮或误导性 debug 节点
- 可以生成并校验微信小游戏构建产物
- 可以明确拿出一个可供测试人员使用的版本

## 4. 当前已知阻塞项

Claude 必须优先处理以下问题，且不得跳序：

1. `BossArena` 的 `BossShieldPhaseController` 初始化报错
2. 通过 `TouchRespawn` 触发重生时，血量恢复链路不稳定
3. `FieldWest` / `FieldRuins` 运行时控制台存在 `StaticVBAccessor` 分配错误
4. 主流程场景仍暴露 `DebugLabel`

建议在 demo 前一并收掉，但优先级低于上面四项的问题：

- `MechanicsLab` 的暂停按钮状态和正式主流程不一致
- 部分视觉节点仍处于占位和正式表现混合态
- 现有提示偏开发视角，不够支撑第一次独立试玩

## 5. Claude 禁止偷懒规则

这一节是强约束。Claude 在执行中必须遵守，不允许自行放宽。

### 5.1 禁止只靠猜测宣布修复

Claude 不允许：

- 只看代码就宣布某个 bug “应该已经修好”
- 只跑 `typecheck` 就宣布运行时问题解决
- 只让一条小测试通过，就宣布整条主流程稳定

Claude 必须：

- 先复现
- 再定位
- 再改动
- 再跑针对性验证
- 再跑整体验证

### 5.2 禁止通过削弱测试获得“通过”

Claude 不允许：

- 删除、跳过、放宽已有测试，只为了让结果变绿
- 把控制台报错改成忽略列表，只为了不在审计里出现
- 把 debug 节点改名来规避检查，而不是清掉对玩家可见的问题

只有在满足以下条件时，才允许修改测试：

- 先证明原测试与真实产品目标不一致
- 修改测试是为了提高验证质量，而不是降低门槛
- 修改后仍然增加了对 demo 目标的保障

### 5.3 禁止只修局部、不做回归

任何涉及下面任一项的改动，都必须做完整回归，而不是只做点状验证：

- 输入
- HUD
- 重生
- 场景切换
- 存档
- Boss
- 微信构建

### 5.4 禁止把“未验证”说成“已验证”

Claude 的最终交付中，必须严格区分：

- 已机器复核通过
- 已平台工具复核通过
- 已人工抽检通过
- 未验证但推测无风险

不能把“没有设备验证”写成“问题应该不大”来替代验证结论。

## 6. Claude 执行总流程

Claude 必须按下面的顺序做事，不能跳阶段。

### 阶段 0：建立基线

在改任何代码之前，Claude 必须先跑并记录当前基线：

```powershell
npm run test:typecheck
npm run test:node
npm run verify:wechat
npx playwright test -c ./playwright.config.mjs tests/demo-playthrough-audit.spec.mjs
```

如果这些命令里已经有失败，Claude 不能直接改代码，必须先明确：

- 哪些失败是当前已知问题
- 哪些失败是新增问题
- 哪些失败会阻塞当前任务

Claude 在执行开始时必须额外创建一份工作日志，建议路径：

- `docs/demo收口执行日志-YYYY-MM-DD.md`

这份日志至少记录：

- 每次复现的命令
- 失败现象
- 改动原因
- 每轮验证结果
- 尚未关闭的风险

### 阶段 1：逐个关闭关键问题

Claude 必须按固定顺序逐个关闭问题：

1. `BossArena` 初始化报错
2. respawn 血量恢复链路
3. `FieldWest` / `FieldRuins` 控制台 critical
4. `DebugLabel` 清理

每个问题都必须执行同一套闭环：

1. 明确复现方式
2. 抓到最小根因
3. 只做解决根因所需的最小改动
4. 跑该问题的针对性验证
5. 跑至少一轮更大范围回归
6. 把结果写入工作日志

### 阶段 1.5：每关闭一个问题后的强制自查

Claude 每完成一个关键问题，都必须立即做一次自查，不能把所有反思拖到最后。

这一步不是“随便写几句感想”，而是一个强制闸门：

- 自查没写，不算完成这个问题
- 自查没有证据，不算完成这个问题
- 自查只写“应该没问题”，不算完成这个问题

Claude 必须在工作日志中为每个问题追加一段 `步骤自查记录`，固定格式如下：

### 步骤自查记录模板

```text
[问题名称]
1. 这一步原目标是什么？
2. 我实际改了哪些文件？
3. 我修掉的是现象、局部补丁，还是根因？依据是什么？
4. 我跑了哪些针对性验证？结果是什么？
5. 我跑了哪些回归验证？结果是什么？
6. 这一步还没有覆盖到什么？
7. 如果现在交付，这一步最大的剩余风险是什么？
8. 我是否满足这一问题的“通过标准”？结论只能写：是 / 否
```

### 步骤自查通过标准

只有同时满足下面条件，才允许进入下一个问题：

- 自查记录已写入工作日志
- 自查里明确列出修改文件
- 自查里明确区分“根因”与“现象”
- 自查里明确列出针对性验证命令和结果
- 自查里明确列出回归验证命令和结果
- 自查结论为 `是`

如果自查结论为 `否`，Claude 不允许继续推进下一个问题，必须先补验证或继续修复。

### 阶段 2：补强“无陪跑可玩”

四个关键问题清掉之后，Claude 才能进入这个阶段。

这一阶段 Claude 需要检查并补强：

- `StartCamp` 到 `BossArena` 的目标提示是否连贯
- `DungeonHub` 的房间完成反馈和 Boss 门状态反馈是否足够明显
- 暂停、重开、返回营地的按钮链路是否完整
- 当前回响、当前目标、当前 HP 是否足够玩家可读

这一阶段的目标不是“好看”，而是“外部测试人员不需要问人就能继续玩”。

### 阶段 3：完成测试分发演练

Claude 不能在代码刚变绿时就宣布任务结束，必须走到“可发测试”的最后一公里。

这一阶段至少要做：

- 重新生成 demo 构建产物
- 重新校验构建产物
- 在微信开发者工具中完成一次主流程 smoke
- 明确版本号、AppID、输出目录、测试分发方式

如果缺少真实 AppID、登录态或设备，Claude 必须明确把这类外部阻塞记录到日志中，但不能因此跳过所有本地与工具链验证。

## 9. 机器复核优先原则

人工抽检不是前置门槛，机器复核才是前置门槛。

Claude 必须先把自己能完成的机器复核全部做满，再进入最后的人类抽检阶段。

本手册把验收状态分成三档：

1. `机器候选通过`
2. `平台候选通过`
3. `最终人工抽检通过`

Claude 至少要拿到 `机器候选通过`，才允许把任务标记为“代码侧完成”。

如果外部条件允许，还应继续推进到 `平台候选通过`。

只有在组织流程要求或条件具备时，才继续做最后的人工抽检。

## 10. 最后人工抽检（后置，不是前置门槛）

人工抽检只放在机器候选通过之后。

它的作用是：

- 给最终分发增加信心
- 发现自动化尚未覆盖的真实设备边角问题

它不是 Claude 在代码执行阶段的首要验收标准。

### 10.1 Cocos 预览抽检

推荐在机器候选通过后做一次人工快速抽检：

- `StartCamp` 进入与首个目标提示
- `FieldWest` 和 `FieldRuins` 进入时控制台状态
- `DungeonHub` 的房间完成反馈
- `BossArena` 的加载、受击、护盾与胜利出口
- 一次真实的死亡 -> 重生 -> 继续推进

通过标准：

- 未发现新的 blocker / critical
- 与机器复核结论一致

### 10.2 微信开发者工具抽检

推荐在机器候选通过后做一次工具侧人工抽检：

- 项目可正常打开
- 可以进入 `StartCamp`
- 摇杆、攻击、回响按钮可用
- 不出现黑屏、卡加载页、输入完全失效
- 至少跑一次主流程 smoke

通过标准：

- 主流程 smoke 可跑
- 未发现机器复核未覆盖的新 blocker / critical

### 10.3 真机抽检

如果具备设备条件，再做最后一层人工抽检：

- 至少 `2` 台设备
- 至少 `1` 次冷启动
- 至少 `1` 次死亡 -> 重生
- 至少 `1` 次退出 -> 重进 -> 读档继续

通过标准：

- 不出现黑屏、卡死、输入失灵、读档断链
- 触屏体验没有明显不可接受的问题

如果设备或账号条件不具备，Claude 必须在最终交付中明确写明：

- 哪一步没法做
- 具体被什么外部条件阻塞
- 下一步由谁提供什么条件才能继续

## 11. 交付证据要求

- 哪些是人工抽检通过

建议 Claude 同时保存以下证据：

- 工作日志文件

## 13. 停止条件与升级条件

Claude 只有在以下情况下才允许停止并寻求人类输入：

- 真实 `AppID`、微信登录态或设备条件缺失
- 修复一个问题会明显引入更大范围行为变化，需要产品判断
- 发现仓库存在与当前任务强冲突的用户未完成改动

除此之外，Claude 不应因为“代码有点复杂”或“已经通过部分测试”就提前结束。

## 14. Demo 出门标准

这份手册把最终状态拆成两个不同层级，防止 Claude 因为最后一层人工条件缺失，就把前面的机器复核价值一并抹掉。

### 13.1 `机器候选通过`

只有满足以下全部条件，Claude 才能把版本标记为 `机器候选通过`：

- `npm run test:typecheck`
- `npm run test:node`
- `npm run test:all`
- `npx playwright test -c ./playwright.config.mjs tests/demo-playthrough-audit.spec.mjs`
- 第 9.1 到 9.4 的机器复核全部通过
- 主流程无阻塞通关 bug
- `BossArena` 不再有初始化 critical
- respawn 血量恢复链路稳定
- `FieldWest` / `FieldRuins` 无当前已知控制台 critical
- 主流程场景无误导性 `DebugLabel`

### 13.2 `平台候选通过`

只有在 `机器候选通过` 基础上，再满足下面条件，Claude 才能把版本标记为 `平台候选通过`：

- `npm run verify:wechat` 通过
- 微信构建产物重新生成并校验通过
- 微信开发者工具 smoke 可跑
- 如有可用的微信模拟器自动化，则自动化结果通过

### 13.3 `建议分发给测试人员`

建议按下面条件标记为 `建议分发给测试人员`：

- 已达到 `平台候选通过`
- 当前回响、目标、Boss 门反馈足够明显
- 测试人员不看文档也能推进主线
- 如果人工抽检条件具备，则人工抽检未发现新的 blocker / critical

如果人工抽检条件暂时不具备，但 `平台候选通过` 已达成，Claude 可以写成：

- `机器与平台复核已通过，具备建议分发条件；最终人工抽检待设备/账号条件补齐后执行`

## 16. 模拟 Claude 专家审阅

审阅角色：

- Claude 项目交付专家
- 关注点是“这份文档能不能约束 Claude 真正把事做完，而不是做出一个看起来合理的半成品”

### 审阅结论

这份文档已经从“方向说明”升级成“执行手册”，可以直接用于约束 Claude 干活。

与普通方案相比，已经明显补强了下面几项：

- 把“先复现再修复”写成强制流程
- 把“每完成一步都必须自查”写成步骤闸门
- 把“不能靠削弱测试变绿”写成硬规则
- 把“机器复核优先，人工抽检后置”写成明确分层
- 把“最终必须交出证据和完成报告”写成交付要求

### 审阅发现的关键加强点

1. 原方案更像人类项目经理的节奏说明，不足以防止 Claude 只修表层问题。

已加强：

- 增加了逐问题闭环
- 增加了禁止偷懒规则
- 增加了停机与升级条件

2. 原方案的验证要求不够细，容易让 Claude 只跑 `typecheck + node`。

已加强：

- 为每个关键问题指定了针对性命令
- 为输入、HUD、场景切换、Boss、存档等改动强制追加 `test:all`
- 将 `demo-playthrough-audit` 升级为收尾前必跑项
- 增加了视觉、响应式、流程、构建四层机器复核

3. 原方案过早把人工复核写成核心门槛，不利于 Claude 先把自己能完成的复核做到极致。

已加强：

- 先定义 `机器候选通过`
- 再定义 `平台候选通过`
- 最后才定义人工抽检
- 即使设备和账号条件缺失，也不会模糊前两层的验收结论

4. 原方案没有强制 Claude 保留执行证据。

已加强：

- 新增工作日志要求
- 新增步骤自查记录模板
- 新增截图和验证记录要求
- 新增最终交付证据清单
- 新增最终完成报告模板

### 专家建议的最终执行口径

如果 Claude 在执行过程中声称“问题应该已经解决”，但没有同时给出：

- 复现证据
- 修改依据
- 针对性验证
- 大范围回归
- 人工或工具链复核

那么默认视为任务没有完成。

这条口径应当被视为本手册的最终解释规则。
