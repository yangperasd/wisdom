## 5. 当前文档里的几个关键问题

### 5.1 把“平台自动化没跑起来”写得过于绝对

[完成报告](/E:/cv5/wisdom/docs/demo完成报告-2026-04-14.md:148) 和 [执行日志](/E:/cv5/wisdom/docs/demo收口执行日志-2026-04-14.md:425) 的问题，不是“记录了平台验证失败”，而是把“没做成 auto-mode 自动化”近似写成了“微信这边没法继续验证”。

这个表述太重了，因为当前审计实际确认到：

- `verify:wechat` 可跑
- `build:wechat` 可跑
- `open:wechat` 可跑
- 本机微信开发者工具已安装，且进程存在

所以更准确的说法应当是：

- `微信自动化模拟器链路未跑通`
- `但微信开发者工具项目拉起能力是存在的`
- `因此不应直接把平台 smoke 说成完全无法验证`

### 5.2 `wechat-simulator.test.mjs` 的 “1/10 PASS” 不应当被当成有效平台证据

[tests/wechat-simulator.test.mjs](/E:/cv5/wisdom/tests/wechat-simulator.test.mjs:102) 的第一个测试只是打印端口状态，然后在 [第 107 行](/E:/cv5/wisdom/tests/wechat-simulator.test.mjs:107) 直接 `assert.ok(true)`。

也就是说：

- 它无论端口开不开都会 PASS
- 这个 PASS 是信息性的，不是功能性的

因此报告里写的：

- [“1/10 PASS，9/10 SKIP”](/E:/cv5/wisdom/docs/demo完成报告-2026-04-14.md:150)

不能被解读成“平台验证完成了 10%”。更准确的解释是：

- `0 个真正的平台运行断言通过`
- `1 个信息性检查完成`
- `9 个真正的平台断言未执行`

### 5.3 “官方 bug” 的归因写得太满

当前我复跑 `node --test tests/wechat-simulator.test.mjs` 时，实际输出是：

- `Auto-mode WebSocket not reachable on port 9420`

这说明当前事实只是：

- `auto-mode 没起来`

但还不能仅凭这一次结果就下结论：

- 一定是 `v2.01.2510290` 的那个已知 bug

可能原因至少包括：

- CLI auto 根本没启动
- 端口没开
- DevTools 当前并不在 auto mode
- 已知版本 bug

所以平台日志里应当区分：

- `已观测到的事实`
- `推测的原因`

而不是把推测原因写成既定事实。

### 5.4 完成报告里有“交付证据”和“仓库实际变更”不完全一致的问题

[完成报告的修改文件清单](/E:/cv5/wisdom/docs/demo完成报告-2026-04-14.md:88) 包含：

- `library/21/217ece9c-aa9f-4460-90b2-f23a5b5f8adb.json`

但这个文件当前不在 `git diff --name-only` 里，也不是 git 跟踪文件。

这不代表报告一定撒谎，但至少说明：

- 它更像“本地运行期辅助证据”
- 不应和真正的仓库交付文件混写成同一层级

这个问题会降低完成报告的可信度。
