# 最终 MVP 推进方案（2026-04-14）

## 1. 目标定义

这里的“最终目标”不是完整复刻《智慧的再现》，而是按照 [方案A-MVP-微信小游戏-类智慧的再现.md](/E:/cv5/wisdom/docs/方案A-MVP-微信小游戏-类智慧的再现.md:1) 收口成一个可交付的微信小游戏 MVP。

最终版本应满足：

- 单次完整通关约 `45 ~ 60` 分钟
- 围绕 `3` 个回响形成完整体验闭环
- 包含 `2` 段野外、`3` 房间迷宫、`1` 个 Boss
- 微信小游戏真机可稳定运行
- 无阻塞通关的严重 bug
- 死亡、离房、回档后不出现软锁

## 2. 当前判断

截至 2026 年 4 月 14 日，项目已经完成了大量底座工作：

- 主流程场景链路已接通
- `3` 个回响、触屏输入、检查点、存档、场景切换已具备
- 迷宫房间、Hub、Boss 场景都已存在
- 微信小游戏构建、校验与打开工具链路可用
- 自动化覆盖已经超过早期“结构 + smoke”阶段

但距离最终 MVP 仍然有明显差距，核心不在“有没有底层框架”，而在：

- 内容深度是否足够支撑 `45 ~ 60` 分钟
- 敌人、房间和 Boss 是否真的形成完整节奏
- 美术、音频、反馈是否达到产品级
- 真机环境是否稳定到可交付

综合判断：

- 距离最终 MVP 约还差 `35% ~ 45%`

## 3. 当前主要缺口

### 3.1 内容缺口

当前主线链路已经存在，但“链路存在”不等于“完整体验成立”。

仍需确认和补足：

- 前 `10` 分钟节奏是否顺畅
- 中段野外和迷宫是否足够支撑体验时长
- `RoomA / RoomB / RoomC` 是否都达到了明确教学目标
- Boss 是否真正验证了 `3` 个回响，而不只是存在一套骨架

### 3.2 战斗与敌人缺口

目标文档要求的是 `4` 种基础敌人，但当前核心敌人行为仍偏通用。

当前 [EnemyAI.ts](/E:/cv5/wisdom/assets/scripts/enemy/EnemyAI.ts:1) 主要还是：

- `Idle`
- `Patrol`
- `Chase`

这更像一个通用敌人底层，而不是已经具备 `4` 类有鲜明差异的敌人设计。

### 3.3 表现层缺口

虽然素材接入层已经存在，但项目仍处于“正式表现 + 占位过渡态并存”的阶段。

当前仍需收口：

- 主流程场景的占位节点清理
- Boss 场景资源接入
- HUD 的产品化表现
- 受击、破坏、开门、拾取等关键反馈

### 3.4 音频缺口

项目已经有音频接口和部分素材目录，但从当前仓库状态看，BGM 侧仍不算完整收口。

最终 MVP 还需要：

- 至少可用的区域 BGM
- Boss 战音乐
- 关键 SFX 统一和去重
- 真机上不刺耳、不丢失、不冲突的播放体验

### 3.5 平台与真机缺口

构建链路已经通了，但“能构建”不等于“能交付”。

最终还需要补足：

- 冷启动体验
- 触屏误触率
- 常见分辨率安全区
- 场景切换卡顿
- 长流程发热与掉帧
- 微信体验版和提审前自测流程

## 4. 总体原则

最终 MVP 阶段继续遵守这些原则：

- 不新增第 `4` 个回响
- 不扩张成开放世界
- 不增加复杂成长系统
- 每一轮都必须有“从头到尾能跑”的版本
- 一旦延期，优先删内容，不补系统

## 5. 推进方案

### 阶段 1：以稳定 demo 为基线

目标：

- 先把“可测 demo”做成稳定基线，再继续扩展和 polish

为什么先做这个：

- 不稳定的基线上继续做内容，会让后面所有回归成本上升
- 真正的最终 MVP 一定建立在“已经可测”的版本上，而不是并行推进两个不稳定目标

阶段完成标准：

- demo 已满足内部测试分发标准
- 当前主线 critical 已清掉
- 微信环境下的最小可玩闭环成立

### 阶段 2：补足内容深度

目标：

- 把当前“场景链路存在”推进到“体验时长和节奏成立”

必做项：

- 重新检查 `StartCamp -> FieldWest -> FieldRuins` 的教学顺序
- 强化 `DungeonHub` 的中继反馈
- 把 `RoomA / RoomB / RoomC` 做成真正的单能力教学房
- 让迷宫推进和 Boss 门解锁形成明确目标感

阶段完成标准：

- 主线中前段不靠开发者解释也能理解
- 迷宫和 Hub 的节奏不再只是“有门有房间”，而是形成连续推进感
- 总时长进入接近目标的区间

### 阶段 3：补足敌人与 Boss 的验证性

目标：

- 让轻战斗和 Boss 真正承担“验证玩家理解”的职责

必做项：

- 区分出更明确的敌人行为类型，而不只是通用追击敌人换皮
- 让普通敌人与机关更好地服务于回响解谜
- 把 Boss 从“第一轮骨架”推进到清晰的多段验证战

Boss 设计重点：

- 必须要求用到 `BombBug`
- 最好要求同时用到 `Box` 或 `SpringFlower`
- 输出期、危险期和读招反馈必须足够明确

阶段完成标准：

- 普通战斗不空洞
- Boss 不再只是概念成立，而是成为真正的结尾关卡

### 阶段 4：表现与音频收口

目标：

- 把项目从“能玩”推进到“像产品”

必做项：

- 清理灰盒与正式资源混用状态
- 给主流程、迷宫、Boss 场景完成第一轮完整表现绑定
- 补齐区域 BGM 和关键 SFX
- 收口 HUD、暂停、结算和关键反馈

阶段完成标准：

- 主流程不再大量暴露占位节点
- 玩家能明显感知到区域差异、回响状态和战斗反馈
- 音频不再是明显缺口

### 阶段 5：真机与交付收口

目标：

- 把最终 MVP 真正推到“可交付”的微信小游戏状态

必做项：

- 真机完整通关回归
- 冷启动与热启动检查
- 常见屏幕比例和安全区检查
- 包体与资源清理
- 体验版、提审前自测和问题回归

阶段完成标准：

- 真机完整通关稳定
- 无明显黑屏、卡死、输入失灵
- 关键场景性能在可接受范围内

## 6. 验收标准

最终 MVP 只有满足以下条件，才算真正达标：

### 内容验收

- 玩家能在 `45 ~ 60` 分钟内完成完整流程
- `3` 个回响在主线中都被多次使用
- 至少 `3` 类谜题模板得到复用
- Boss 明确要求使用回响机制

### 稳定性验收

- 无阻塞通关 bug
- 死亡、离房、回档不会导致软锁
- 场景切换、存档和进度恢复稳定

### 平台验收

- 微信小游戏真机稳定运行
- 包体与资源不出现失控膨胀
- 关键输入在主流手机上可接受

### 表现验收

- HUD、提示、暂停、结算达到可交付最低标准
- 区域表现和 Boss 表现不再明显处于原型态
- 音频反馈基本完整

## 7. 推荐工期

在“先完成 demo 基线”的前提下，最终 MVP 建议按 `3 ~ 5` 个中冲刺推进。

单人视角的粗略预估：

- `4 ~ 8` 周

影响工期最大的变量有两个：

- 正式素材和音频到位速度
- 真机环境暴露出来的问题数量

## 8. 推荐顺序

推荐严格按下面顺序推进：

1. 先完成“给测试人员的 demo”
2. 再补主线节奏与内容深度
3. 再补敌人与 Boss 验证性
4. 再做表现与音频收口
5. 最后做真机与交付收口

这个顺序的原因是：

- 先测，能最早暴露真实问题
- 先稳，再 polish，返工更少
- 先闭环，再扩表现，性价比最高

## 9. 当前建议

从今天的节点看，最合理的路径不是同时硬推两个目标，而是：

- 先用最短时间把 demo 修到可以发测试
- 立刻吃掉第一轮真实试玩反馈
- 再用这些反馈决定最终 MVP 阶段优先修“内容节奏”还是“真机体验”

最终 MVP 的成功，不取决于再加多少系统，而取决于是否能把现在这套竖切真正收成一款完整、稳定、可玩完的微信小游戏。
