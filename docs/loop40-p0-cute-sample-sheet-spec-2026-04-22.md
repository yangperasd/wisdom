# Loop40 P0 Cute Sample Sheet Spec

日期：2026-04-22

## 目标

在导入任何新的 tile / wall bitmap 前，先用一张极小样张验证《智慧的再现》方向的视觉语法：明亮、温暖、可爱、圆润、玩具感、低噪声、路径清楚。

这份规格不批准最终美术，不要求现在生成图片，也不改变主包策略。它只定义：如果下一步要做样张，样张必须长什么样、测什么、怎么失败。

## 覆盖范围

P0 必须覆盖五个 `must-redo` key：

| Key | 样张目标 | 首发影响 |
| --- | --- | --- |
| `outdoor_wall_standard` | 标准边界 / 矮墙 / 场景边缘语言 | `StartCamp`, `BossArena` |
| `outdoor_wall_broken` | 破损边界，但不能写实阴暗 | `FieldRuins` |
| `outdoor_wall_cracked` | 可破坏 / 裂纹提示，但必须卡通低压迫 | `FieldRuins`, `BossArena` |
| `outdoor_path_cobble` | 主路径 tile，必须 3 秒内读路 | `StartCamp`, `FieldWest`, `FieldRuins`, `DungeonRoomA`, `DungeonRoomC` |
| `outdoor_ground_flowers` | 温暖点缀地表，不能变成脏绿噪点 | `StartCamp`, `FieldWest`, `FieldRuins`, `DungeonRoomB` |

## 样张最小交付

样张只需要证明风格语法，不需要覆盖完整场景。

| 交付物 | 要求 |
| --- | --- |
| Contact sheet | 一张合成预览图，展示全部样张并标 key |
| 单 tile 预览 | 每个 key 至少 1 个主版本，最多 2 个变体 |
| 场景小拼图 | `StartCamp` 入口、`FieldWest` 路径、`DungeonHub` 入口、`BossArena` 墙体各一块拼贴预览 |
| 包体估算 | 导入前估算新增压缩体积，目标 `<= 256 KiB`，超过必须拆分或缩减 |
| 风格评分 | 每个 key 按 8 项评分，总分低于 `12/16` 不得导入工程 |

建议尺寸：

| 类型 | 建议 |
| --- | --- |
| 单 tile | `128x128` 或 `256x256`，优先小尺寸 |
| Contact sheet | 不超过 `1024x1024` |
| 颜色数量 | 控制色板，避免高频噪点 |
| 边缘 | 可无缝重复，但优先读路和圆润轮廓 |

## 风格评分

每项 `0/1/2` 分。

| 维度 | `2` 分标准 | `0` 分失败例 |
| --- | --- | --- |
| 明度 | 第一眼明亮、可亲近 | 黑灰压屏 |
| 暖度 | 米色、暖绿、浅褐、柔黄为主 | 冷蓝灰、冷金属 |
| 低噪声 | 大色块清楚，纹理不抢角色 | 脏绿噪点、碎纹太密 |
| 玩具感 | 像小模型 / 纸雕 / 黏土玩具 | 写实石材、恐怖地牢 |
| 圆润轮廓 | 边缘柔和，破损也不尖锐 | 哥特尖刺、利刃裂纹 |
| 路径可读 | 3 秒内看懂可走方向 | 背景和路径糊成一片 |
| HUD 兼容 | 不和温暖程序化 HUD 打架 | 大金边、冷硬 UI 感 |
| 角色可见度 | 玩家和交互点不被吞没 | 地表比角色更抢眼 |

## 禁止项

命中任一项直接退回：

| 禁止项 | 原因 |
| --- | --- |
| 黑灰大底 | 会滑回黑暗地牢 |
| 脏绿噪点 | 当前截图的主要丑感来源之一 |
| 写实裂纹 | 和可爱玩具感冲突 |
| 尖锐哥特 | 会改变游戏气质 |
| 血腥压迫 | 微信小游戏首发不适合 |
| 冷硬金属 | 和温暖营地/童话废墟冲突 |
| 过重金边 | 会重新制造 HUD 混搭 |
| 文本烘进贴图 | 不利本地化和 HUD 统一 |
| AI 伪影 | 变形、脏边、纹理糊、局部风格断裂 |

## 预览位置

样张通过导入前审查后，才允许考虑进工程。预览应优先使用最小场景拼贴，不直接全场替换。

| 预览 | 目的 |
| --- | --- |
| `StartCamp` 入口拼贴 | 验证首屏是否更温暖可读 |
| `FieldWest` 路径拼贴 | 验证野外路径是否清楚 |
| `FieldRuins` 废墟拼贴 | 验证遗迹是否可爱而不压迫 |
| `DungeonHub` 门厅拼贴 | 验证地牢入口是否摆脱黑暗 |
| `BossArena` 边界拼贴 | 验证终局空间是否有仪式感但不恐怖 |

不建议第一轮直接替换 `DungeonRoomA/B/C` 全场。三房间应在样张语法确认后，再分别做箱子、弹花、炸虫题眼规格。

## 包体门禁

任何新 bitmap 进入工程前必须满足：

| 检查 | 标准 |
| --- | --- |
| 主包预算 | 仍保持 `<= 4 * 1024 * 1024 bytes` |
| 预警线 | 新增后仍尽量低于 `3.7 MiB` |
| 增量预算 | P0 样张阶段目标新增 `<= 256 KiB` |
| 远程脚本 | 必须为 `0` |
| 新资源归属 | 首屏必需才进主包，否则优先分包或暂不导入 |

如果为了样张导入资源导致主包接近预警线，应立即停止导入，回到 contact sheet 评审。

## 文件命名建议

如果样张后续获批进入工程，建议先使用明确的候选命名，不直接覆盖当前 key。

`candidate` 只表示“可进入评审”，不表示“可导入工程”、不表示“可改正式绑定”、也不表示“已成为最终美术”。

| Key | 候选文件名 |
| --- | --- |
| `outdoor_wall_standard` | `cute_wall_standard_candidate_a.png` |
| `outdoor_wall_broken` | `cute_wall_broken_candidate_a.png` |
| `outdoor_wall_cracked` | `cute_wall_cracked_candidate_a.png` |
| `outdoor_path_cobble` | `cute_path_cobble_candidate_a.png` |
| `outdoor_ground_flowers` | `cute_ground_flowers_candidate_a.png` |

候选图通过验收前，不允许改写 `asset_binding_manifest_v2.json` 的正式绑定。

## 验收流程

1. 制作 contact sheet。
2. 对五个 key 做 8 项评分。
3. 在不导入工程的情况下做拼贴预览。
4. 人类确认是否符合可爱风北极星。
5. 估算压缩后体积。
6. 如需导入，先作为候选资源进入分支并跑包体校验。
7. 只有截图、评分、包体、首局读路都通过后，才考虑更新正式绑定。

## 失败样例

| 现象 | 判定 |
| --- | --- |
| 看起来更精细，但更像写实地牢 | 失败 |
| 颜色更亮，但地表噪声更密 | 失败 |
| 墙体好看，但玩家/路径被吞没 | 失败 |
| 需要靠文字才能知道哪里能走 | 失败 |
| 单图漂亮，但放进场景和 HUD 打架 | 失败 |
| 资源体积过大，需要牺牲主包余量 | 失败 |

## 结论

P0 样张阶段只做“小而硬”的风格验证。它的成功标准不是多画几张图，而是证明五个 must-redo key 可以在低包体成本下，把当前场景从暗、噪、原型感，拉向明亮、温暖、可爱、圆润、玩具感。
