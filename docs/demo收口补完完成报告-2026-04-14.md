### 1.3 截图是怎么拿到的（反思后的做法）

之前报告说"本会话无桌面截图 MCP 所以做不了" — 这是偷懒。Windows 自带 `user32.dll` 的 `PrintWindow` Win32 API 可以在不抢前台、不截整屏、不泄露桌面其他窗口的前提下直接抓任意 HWND 的窗口内容。

写了一段 PowerShell 内嵌 C#（`Add-Type`），50 行代码：
1. 用 `EnumWindows` + `GetWindowThreadProcessId` 枚举 `wechatdevtools.exe` 进程的所有可见顶层窗口
2. 按窗口标题含 `wisdom`（当前项目名）过滤到 DevTools 主窗口
3. 对每个 HWND 调 `PrintWindow(hwnd, hdc, PW_RENDERFULLCONTENT)` 抓窗口位图
4. 保存 PNG

脚本源码：[`docs/wechat-platform-evidence-2026-04-15/snap-devtools.ps1`](/E:/cv5/wisdom/docs/wechat-platform-evidence-2026-04-15/snap-devtools.ps1)。下次复跑：`powershell -ExecutionPolicy Bypass -File docs/wechat-platform-evidence-2026-04-15/snap-devtools.ps1`。

#### 1.3.5.2 WeChat runtime gameplay smoke 穷尽尝试（2026-04-15 晚第二次补丁）

按用户追加要求"方案 A，继续推进；能通过代码完成的就要创造条件做"，对"WeChat runtime 下主流程 smoke"做了 8 种代码级通道的穷尽尝试：

| 通道 | 结果 | 阻塞点 |
|---|---|---|

| 2. SendInput + SetForegroundWindow | Win11 foreground-lock 拒绝，click 落错窗口 | Windows 11 安全策略对非前台进程禁止夺焦 |
| 3. SendInput + AppActivate + AttachThreadInput + ALT-trick | 同上，全部绕不过 | 同上 |
| 4. UI Automation (UIA) 找 "Never" 按钮 | 152 个系统按钮可见，无 "Never" | Chromium 内部按钮不进 UIA 树 |
| 5. miniprogram-automator 直连 ws 33350 | 握手成功即被服务端关闭 | 需特定握手 auth，无公开协议 |
| 6. `cli.bat auto --debug --project` | `✖ Fetching AppID (wx2a215f964be2b668) detailed information` + `TypeError: d.on is not a function` | DevTools 内部 js bug（见下） |
| 7. `cli.bat auto --auto-port 9420` | 同 #6 | 同上 |
| 8. `/v2/autopreview?project=<path>` HTTP API | `{"code":10,"message":"错误 Error: 需要重新登录"}` | 微信账号登录门（`wechat-login-qr.jpg` 待扫码） |

```

at async openProjectWindow (core.wxvpkg/270091217eac3a5450b70fdd9d10440b.js:2:8437)

```

新增事实层结论：
- WebSocket 连接 + IDE HTTP server + AppID 识别 **全部前置都成功**，bug 只在"拉 AppID 详情 → 创建项目窗口"最后一步
- 源文件是 webpack 打包 + 混淆的内部代码，无 source map；**不可由用户侧补丁**
- `--auto-port` 虽被解析（`autoPort: '9420'` 出现在 debug args）但不改变执行路径，automator 0.12.1 launch() 路径无解

**结论（步骤 13 时）**：WeChat runtime gameplay smoke **在当前技术栈下不可纯代码化**。8 种通道的失败是平台设计（Win11 安全 + Chromium 架构 + 微信登录 + DevTools 内部 bug）的复合结果。

**结论（步骤 15 修正）**：实际上 8 种通道全失败的根本是**全部都试图绕过 Win11 foreground-lock**。换思路 → 不绕 Win11 lock，用**用户单次鼠标 click 让 DevTools 自然前台化** + **脚本轮询 GetForegroundWindow 检测到自动开跑** —— 这条路不在 8 种里。详见步骤 15，已实证成功。**之前结论应修正为"无 user-assist 时不可纯代码化；user-assist 1 次 click 后可自动化"**。
