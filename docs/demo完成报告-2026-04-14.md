## 3. 本次关闭的问题

### 问题 1：`BossArena` 初始化报错

- 根因：`BossArena.scene` 中 `BossShieldPhaseController.shieldTarget` 被序列化为 `cc.Node`（`__id__ 223`，BossShield-Closed 节点），而非该节点上的 `BreakableTarget` 组件（`__id__ 239`）。代码 `this.shieldTarget?.events.on(...)` 命中 Node 上不存在的 `events`，触发 `TypeError: Cannot read properties of undefined (reading 'on')`。生成器 `tools/generate-week2-scenes.mjs` 直接写 scene JSON，绕过了 `@property(BreakableTarget)` 的编辑器类型校验。
- 解决方式：把 scene 里 `shieldTarget.__id__` 由 223 改为 239（指向 BreakableTarget 组件），同步修 library 缓存让 preview 立刻生效；生成器把 `addComponent(...BreakableTarget...)` 的返回值保存为 `bossShieldBreakable`，并把 `shieldTarget: ref(bossShieldClosed.nodeId)` 改成 `shieldTarget: ref(bossShieldBreakable)`，从源头杜绝。
- 影响文件：
  - `assets/scenes/BossArena.scene`
  - `library/21/217ece9c-aa9f-4460-90b2-f23a5b5f8adb.json`
  - `tools/generate-week2-scenes.mjs`

### 问题 2：respawn 血量恢复链路

- 根因：自动死亡走 `HealthComponent → onHealthDepleted → 仅当 checkpoint 存在才调 respawnAt（resetFull 在这里）`；手动 `TouchRespawn` 与键盘 `R` 只走 `GameManager.requestRespawn()`，**没有任何代码再调 `respawnAt`**，所以 HP 永远不会重置。HP 重置职责被绑死在 `onHealthDepleted` 的第二段代码里，手动按钮分支没法触达。
- 解决方式：让 `PlayerController` 监听 `GAME_EVENT_RESPAWN_REQUESTED`，在 `onRespawnRequested` 里统一 `respawnAt`（重置位置 + HP + attackTimer + forcedMove）；`onHealthDepleted` 简化为只调 `GameManager.requestRespawn()`。两条路径都收敛到同一事件、同一处理函数。`killPlayer` 测试辅助加 `disableAutoRespawn` 选项，以保留对"已死状态"的隔离测试能力。
- 影响文件：
  - `assets/scripts/player/PlayerController.ts`
  - `tests/helpers/playwright-cocos-helpers.mjs`
  - `tests/checkpoint-respawn.spec.mjs`（含新增 `manual respawn restores player health…` 用例）
  - `tests/demo-playthrough-audit.spec.mjs`（`Player death and respawn` 由软日志改为硬断言）
  - `tests/error-recovery.spec.mjs`（BUG#2 用 `disableAutoRespawn` 隔离）

### 问题 3：`FieldWest` / `FieldRuins` `StaticVBAccessor` critical

- 根因：FieldBackdrop（1840×560）/ RuinsBackdrop（1960×560）通过 `SceneDressingSkin` 把 32×32 的地表纹理 TILED 铺开，分别产生 1,044 / 1,116 个 tile，单次绘制需 4,176 / 4,464 vertex（约 150KB / 161KB）；Cocos 默认 `cc.macro.BATCHER2D_MEM_INCREMENT = 144`KB，单次最多 `floor(144*1024/36) = 4,096` vertex，**两个 backdrop 都恰好越线**，`allocateChunk` 拒绝并打 critical，相应 backdrop 静默不绘制。一开始把 macro 写在 `PhysicsBootstrap.onLoad` 里没生效，因为 `Batcher2D.switchBufferAccessor` 在第一次需要某 stride accessor 时就构造并冻结 `_vCount`，scene 组件 onLoad 已经晚了。
- 解决方式：把 `macro.BATCHER2D_MEM_INCREMENT = 256` 写到 `PhysicsBootstrap.ts` 的**模块顶层**（脚本类注册时执行，远早于第一次 sprite render），单次可分配 `floor(256*1024/36) = 7,281` vertex（≈1,820 tile），覆盖现有所有 backdrop。这是 Cocos 引擎自己开放的批处理预算调节旋钮（最大允许 2,303 KB），不是绕过控制台错误。
- 影响文件：
  - `assets/scripts/core/PhysicsBootstrap.ts`

### 问题 4：主流程 `DebugLabel` 清理

- 根因（双层）：
  1. 静态层：`MechanicsLab.scene` 的 DebugLabel 节点和 3 个运行时 prefab（`EchoBox`、`EchoSpringFlower`、`ArrowProjectile`）的 DebugLabel 子节点都默认 `_active=true`。
  2. **运行时层（更深）**：`SpriteVisualSkin.setPlaceholderLabelVisible(node, visible)` 会遍历所有带 Label 的子节点并强制 `child.active = visible`；在 placeholder 模式（spriteFrame 未绑定）下 `visible=true`，会把 prefab 里设的 `_active=false` 重新点亮 — 仅靠静态修复无法根治。
- 解决方式：
  - 4 个资产里的 DebugLabel 节点 `_active: true → false`
  - 生成器 `generate-mechanics-lab.mjs` 把 DebugLabel 类型的 `addPrefabLabel` / `addLabeledNode` 默认 inactive
  - **核心根因**：在 `setPlaceholderLabelVisible` 里加 `isDeveloperOnlyLabel(child)` 例外，对 `debug` / `dev-` / `test-` 命名的子节点始终保持 inactive
  - 把 `demo-playthrough-audit.spec.mjs` 的 `walkLabels` / `walkDebug` 改用 `node.activeInHierarchy`（之前用 `node.active` 是局部 `_active`，不反映累积可见状态），并对 Label 加 `enabled !== false` 过滤
  - `mechanics-lab.scene.test.mjs` 的 DebugLabel 断言由 `_active === true` 改为 `false`，加注释说明语义
- 影响文件：
  - `assets/prefabs/EchoBox.prefab` / `EchoSpringFlower.prefab` / `ArrowProjectile.prefab`
  - `assets/scenes/MechanicsLab.scene`
  - `tools/generate-mechanics-lab.mjs`
  - `assets/scripts/visual/SpriteVisualSkin.ts`
  - `tests/demo-playthrough-audit.spec.mjs`
  - `tests/mechanics-lab.scene.test.mjs`

## 7. 剩余风险

1. **真实小游戏 AppID 未替换**。当前 `project.config.json.appid = wx2a215f964be2b668` 是仓库默认占位，不能直接走微信审核或真机预览。下次分发前必须由测试方/产品方提供真实 AppID 并通过 `WECHATGAME_APPID` 重新构建。
2. **微信开发者工具自动化条件缺失**。`wechat-simulator.test.mjs` 因 DevTools v2.01.2510290 的 `cli.bat auto` 官方 bug，10 个用例只能跑 1 个端口探活，其余 9 个 skip。需要降级（或升级到无 bug 版本）才能拿到完整"平台候选通过"。
3. **真机/真键盘抽检为零**。所有验证都在 chromium headless + Cocos preview 完成；KeyboardInputDriver 的 R 重生路径只在结构上被覆盖（共享 `GameManager.requestRespawn()` 入口），未做端到端真键盘按键回归。
4. **目标提示语言不统一**。StartCamp、FieldWest 是中文；FieldRuins → BossArena 是英文。每条提示自己都是 actionable 的，但中文测试者会有"语言切换感"。属阶段 5.2 列入禁止的"纯审美 polish"，本轮没动；最终 MVP 阶段建议统一翻译成中文。
5. **`MechanicsLab` 暂停按钮非主流程**。MechanicsLab 不在 demo 主流程范围（StartCamp → BossArena），但 audit 仍报 1 个 cosmetic（`Pause button is not active`）。属方案 4 的"低优先级"项，不建议本轮处理。
6. **生成器与 scene 文件可能再分叉**。问题 1 的 BossArena 修复同时改了 scene 文件和生成器，但其他场景的同类错位（如果有）目前没法静态扫出来 — 只覆盖了 `shieldTarget`。如果未来再生成 scene 出现别的"Node 而非 Component"序列化失误，需要单独识别。
7. **库缓存与源 scene 偶发不同步**。问题 1 修复时手动同步过 `library/21/...`，但下次 Cocos Creator 在编辑器里打开 BossArena 时会从源 scene 重新生成 library 缓存，源 scene 已正确，所以方向是对的；不放心可以在 Cocos Creator 第一次打开时确认 BossArena `BossShieldControllerNode` 的 `Shield Target` 属性指向 `BossShield-Closed` 上的 `BreakableTarget` 组件，不是节点本身。
